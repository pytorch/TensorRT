.. _stream_management:

Graph-Level Stream Management
==============================

This document describes the design and implementation of graph-level stream
management in Torch-TensorRT, enabling concurrent execution of independent TRT
engines on separate CUDA streams and SM partitions (Green Contexts).

----

Problem Statement
-----------------

A partitioned ``GraphModule`` produced by ``torchtrt.compile()`` contains
multiple ``call_module`` nodes, each wrapping a ``TorchTensorRTModule``.
Without explicit stream management:

1. All engines execute sequentially on the default CUDA stream — independent
   subgraphs cannot overlap.
2. There is no first-class representation of cross-stream data dependencies
   (fan-in patterns like vision + language → action head).
3. Stream binding via mutable per-engine state is fragile and invisible to
   graph tools (profilers, serialisers, ``torch.compile``).

----

Goals
-----

1. Stream management is **explicit in the FX graph** — baked in at
   post-compile time, visible to all graph tools.
2. **DAG-aware barrier insertion** — ``sync_streams`` pairs are derived
   automatically from data dependencies; fan-in patterns are handled
   correctly with no user involvement.
3. **Conditional fork** — if the caller is already on a non-default stream
   (e.g. a Green Context stream), the graph runs on that stream; a fork only
   occurs when the caller is on the default stream.
4. **User-provided streams** — supports ``torch.cuda.ExternalStream`` wrapping
   Green Context streams or any other externally-managed stream.
5. **RAII semantics** — ``stream_plan()`` yields a new ``GraphModule``; the
   original is untouched; stream attributes are released on exit.
6. **Post-compilation** — submodule names (``_run_on_acc_*``) are generated by
   the partitioner and are not known until after compilation; the pass works
   entirely on the compiled graph.

Non-Goals
---------

- Changing the compilation pipeline — this is a pure runtime / post-compile
  pass.
- NCCL + Green Context interaction — deferred to a follow-up.

----

Why Not ``torch.ops.streams``?
-------------------------------

PyTorch's ``torch.ops.streams.fork`` / ``join`` ops (``pytorch/pytorch#167152``)
are incompatible with our use case for one fundamental reason.

``fork(from_index, to_index)`` requires **compile-time integer indices** baked
into the graph at dynamo trace time — they index into a table that dynamo
populates when it encounters ``StreamVariable`` objects during tracing.

Our "from" stream (the caller's current stream at graph entry) is a **runtime
value**. It changes per call — the caller may be on the default stream one
invocation and on a Green Context stream the next.  ``enter_compute_stream``
captures ``torch.cuda.current_stream()`` **at runtime** and returns its raw
handle as an ``int`` that flows as a data value to ``exit_compute_stream``.
There is no way to express that with ``torch.ops.streams.fork`` since both
indices must be statically known when the graph is built.

Architecture
------------

Overview::

    torchtrt.compile(net, inputs=[...])
            │
            ▼
      partitioned GraphModule
      ┌─────────────────────────────────┐
      │ placeholder: x                  │
      │ call_module[_run_on_acc_0]       │  ← TorchTensorRTModule
      │ call_module[_run_on_acc_1]       │  ← TorchTensorRTModule
      │ output                          │
      └─────────────────────────────────┘
            │
            ▼  torchtrt.runtime.stream_plan(model, streams=[s0, s1])
            │
      planned GraphModule  (new object, submodules shared)
      ┌──────────────────────────────────────────────────────────┐
      │ placeholder: x                                           │
      │ %cid = enter_compute_stream(s0_raw, dev)                 │ ← conditional fork
      │ %tok0 = set_stream(s0_raw, dev)                          │
      │ sync_streams(%cid, s0_raw, dev)                          │ ← caller→s0 barrier
      │ %r0 = call_trt_with_token(%tok0, _run_on_acc_0, x)       │ ← runs on s0
      │ sync_streams(s0_raw, s1_raw, dev)                        │ ← s0→s1 barrier
      │ %tok1 = set_stream(s1_raw, dev)                          │
      │ %r1 = call_trt_with_token(%tok1, _run_on_acc_1, %r0)     │ ← runs on s1
      │ sync_streams(s1_raw, %cid, dev)                          │ ← s1→caller barrier
      │ exit_compute_stream(%cid, dev)                           │ ← restore caller
      │ output: %r1                                              │
      └──────────────────────────────────────────────────────────┘

VLA fan-in example::

    image ──→ [vision_encoder]  stream_A ──┐
                                            ├──→ [action_head]  stream_C
    text  ──→ [language_model]  stream_B ──┘

The action head has predecessors on two different streams. The pass emits two
``sync_streams`` calls before it — one from stream_A, one from stream_B.
No user involvement required.

Part 1: Stream Control Ops
---------------------------

Four ops in the ``tensorrt`` namespace manage CUDA stream state:

+---------------------------+-------------------------------------------------------+
| Op                        | Semantics                                             |
+===========================+=======================================================+
| ``enter_compute_stream``  | If caller is on default stream, switch to the given   |
|                           | compute stream. Returns caller's raw handle as token. |
+---------------------------+-------------------------------------------------------+
| ``set_stream``            | Unconditionally switch to the given stream.           |
|                           | Returns ``stream_id`` as an int token.                |
+---------------------------+-------------------------------------------------------+
| ``sync_streams``          | Record a CUDA event on ``src``, make ``dst`` wait.    |
|                           | Returns ``dst_stream_id`` as token.                   |
+---------------------------+-------------------------------------------------------+
| ``exit_compute_stream``   | Restore the current CUDA stream to ``caller_id``.     |
+---------------------------+-------------------------------------------------------+

C++ registration (``core/runtime/stream_ops.cpp``)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Schemas and eager CUDA implementations are registered in C++ via
``TORCH_LIBRARY_FRAGMENT`` and ``TORCH_LIBRARY_IMPL(tensorrt, CompositeExplicitAutograd)``.
They are compiled into ``libtorchtrt.so`` and registered when the library loads.

.. code-block:: cpp

    TORCH_LIBRARY_FRAGMENT(tensorrt, m) {
        m.def("enter_compute_stream(int stream_id, int device_index) -> int");
        m.def("exit_compute_stream(int caller_stream_id, int device_index) -> ()");
        m.def("set_stream(int stream_id, int device_index) -> int");
        m.def("sync_streams(int src_stream_id, int dst_stream_id, int device_index) -> int");
    }

    TORCH_LIBRARY_IMPL(tensorrt, CompositeExplicitAutograd, m) {
        m.impl("enter_compute_stream", enter_compute_stream);
        m.impl("exit_compute_stream", exit_compute_stream);
        m.impl("set_stream", set_stream);
        m.impl("sync_streams", sync_streams);
    }

Python fallback (``py/torch_tensorrt/dynamo/runtime/_stream_ops.py``)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

When ``ENABLED_FEATURES.torch_tensorrt_runtime`` is ``False`` (Python-only
builds without the compiled C++ runtime), equivalent implementations using
``torch.cuda.ExternalStream`` are registered from Python:

.. code-block:: python

    if not ENABLED_FEATURES.torch_tensorrt_runtime:
        _schema_lib = torch.library.Library("tensorrt", "FRAGMENT")
        _schema_lib.define("enter_compute_stream(int stream_id, int device_index) -> int")
        # ... other schemas

        _impl_lib = torch.library.Library("tensorrt", "IMPL", "CompositeExplicitAutograd")
        _impl_lib.impl("enter_compute_stream", _enter_compute_stream_py)
        # ... other implementations

Meta kernels (``py/torch_tensorrt/dynamo/runtime/meta_ops/register_meta_ops.py``)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Abstract / fake implementations for ``torch.compile`` tracing are registered
alongside ``fake_tensorrt_execute_engine``, following the same convention:

.. code-block:: python

    @torch.library.register_fake("tensorrt::enter_compute_stream")
    def fake_enter_compute_stream(stream_id: int, device_index: int) -> int:
        return 0

    @torch.library.register_fake("tensorrt::set_stream")
    def fake_set_stream(stream_id: int, device_index: int) -> int:
        return stream_id

    # ... etc.

Ordering guarantee
~~~~~~~~~~~~~~~~~~

Two complementary mechanisms ensure stream control ops execute before the TRT
engine calls that depend on them:

1. **``EffectType.ORDERED``** — registered on all four stream ops via
   ``_register_effectful_op(op, _EffectType.ORDERED)``. AOT Autograd wraps
   them with ``with_effects()``, threading dummy token tensors between
   consecutive ORDERED ops. This prevents DCE and reordering between stream
   control ops, and works correctly whether the implementation comes from C++
   or the Python fallback.

2. **``call_trt_with_token`` HOP** — ``call_module`` (TRT engine) nodes have
   no data dependency on preceding ``call_function`` stream-control nodes, so
   a scheduler could legally reorder them despite ``EffectType.ORDERED`` on the
   stream ops. The FX pass replaces each TRT ``call_module`` with a
   ``call_trt_with_token`` higher-order operator whose **first argument is the
   int token** returned by the immediately preceding ``set_stream`` or
   ``sync_streams`` node:

   .. code-block:: python

       %tok = set_stream(stream_id, device)          # returns stream_id as int
       %out = call_trt_with_token(%tok, submod, *inputs)  # tok is a hard dep

   This creates an explicit data-flow edge. Any scheduler that respects
   data-flow order is forced to execute stream setup before the engine call.

Part 2: StreamPlan API
-----------------------

File: ``py/torch_tensorrt/runtime/stream_plan.py``

.. code-block:: python

    @contextmanager
    def stream_plan(
        gm: torch.fx.GraphModule,
        streams: list[torch.cuda.Stream] | None = None,
        hints: dict[str, torch.cuda.Stream] | None = None,
    ) -> Iterator[torch.fx.GraphModule]:
        """RAII context manager. Yields a planned GraphModule."""

    def apply_stream_plan(
        gm: torch.fx.GraphModule,
        streams: list[torch.cuda.Stream] | None = None,
        hints: dict[str, torch.cuda.Stream] | None = None,
    ) -> torch.fx.GraphModule:
        """Permanent (non-RAII) variant."""

Plan validation
~~~~~~~~~~~~~~~

+-------------------------------------+-------------------------------+
| Condition                           | Error                         |
+=====================================+===============================+
| Hint names a node not in graph      | Unknown node                  |
+-------------------------------------+-------------------------------+
| ``streams`` count != TRT node count | Count mismatch                |
+-------------------------------------+-------------------------------+
| Positional + hint conflict          | Conflicting assignment        |
+-------------------------------------+-------------------------------+
| Stream device != engine device      | Device mismatch               |
+-------------------------------------+-------------------------------+
| CUDA graphs enabled                 | Mutex violation               |
+-------------------------------------+-------------------------------+

Everything else — fan-in barriers, stream transitions, multi-stream
convergence — is handled automatically. The pass never throws for
topological reasons.

Part 3: The FX Pass
--------------------

File: ``py/torch_tensorrt/runtime/stream_plan.py`` (``_apply_stream_plan``)

The pass creates a new ``GraphModule`` that shares TRT engine submodules with
the original (no re-build) but owns a new graph with stream ops inserted.
Stream objects are stored as module attributes (``_trt_stream_N``) to keep
them alive for the lifetime of the planned module; their raw integer handles
are passed as literal int args to the stream ops.

Algorithm
~~~~~~~~~

**Step 1 — Barriers at cross-stream edges.**
For each TRT node N, for each unique source stream among its direct
predecessors that differs from N's stream, insert one
``sync_streams(src_raw, dst_raw, device)`` before N.
Predecessors on the caller stream are deferred until Step 3.

**Step 2 — Stream switches.**
Walk in execution order tracking the current stream. Emit
``set_stream(stream_raw, device)`` whenever the assigned stream changes.
``set_stream`` returns its ``stream_id`` as an int token consumed by
``call_trt_with_token`` in Step 6.

**Step 3 — Graph entry.**
Insert ``enter_compute_stream(primary_raw, device)`` before the first compute
node. It returns the caller's raw stream handle as ``caller_id_node``.
Patch the deferred caller-stream barriers from Step 1 using this node.

**Step 4 — Graph exit.**
Insert ``sync_streams(last_raw, caller_id_node, device)`` then
``exit_compute_stream(caller_id_node, device)`` before the output node.

**Step 5 — Lint and recompile.**
``graph.lint()`` then ``planned.recompile()``.

**Step 6 — Replace ``call_module`` with ``call_trt_with_token``.**
Scan each TRT ``call_module`` node backward to find the most recent
stream-control node (``set_stream``, ``sync_streams``, or
``enter_compute_stream``). Replace the ``call_module`` with:

.. code-block:: python

    submod_node = graph.get_attr(n.target)          # FX graph owns reference
    new_node = graph.call_function(
        call_trt_with_token,
        args=(last_ctrl, submod_node, *n.args),     # last_ctrl is int token
    )

Part 4: Debugger Integration
-----------------------------

Stream plan visualisation lives in ``py/torch_tensorrt/dynamo/debug/_stream_viz.py``
as part of the debugger subsystem (requires ``pip install '.[debug]'`` for
``pydot``).

.. code-block:: python

    from torch_tensorrt.runtime import print_stream_plan, show_stream_plan

    # Text summary to stdout
    print_stream_plan(planned_gm, plan)

    # PNG to DEBUG_LOGGING_DIR/stream_plan.png
    show_stream_plan(planned_gm, plan)

    # PNG into the active debug session's logging directory
    with Debugger(logging_dir="/tmp/run1") as dbg:
        planned = apply_stream_plan(model, streams=[s0, s1])
        dbg.dump_stream_plan(planned_gm, plan)
        # writes /tmp/run1/stream_plans/stream_plan.png

``stream_plan_dot(gm, plan)`` returns a ``pydot.Dot`` object for custom
rendering (call ``.write_svg()``, ``.write_png()``, etc. directly).

Part 5: User-Facing Examples
-----------------------------

Single compute stream (default case)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

.. code-block:: python

    import torch_tensorrt

    model = torch_tensorrt.compile(net, inputs=[...])

    s = torch.cuda.Stream()
    with torch_tensorrt.runtime.stream_plan(model, streams=[s]) as m:
        out = m(x)

Green Context multi-graph (VLA)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

.. code-block:: python

    vision_stream   = torch.cuda.ExternalStream(cuGreenCtxStreamCreate(ctx_A))
    language_stream = torch.cuda.ExternalStream(cuGreenCtxStreamCreate(ctx_B))
    action_stream   = torch.cuda.ExternalStream(cuGreenCtxStreamCreate(ctx_C))

    model = torch_tensorrt.compile(vla_net, inputs=[image_inp, text_inp])

    with torch_tensorrt.runtime.stream_plan(
        model,
        streams=[vision_stream, language_stream, action_stream],
    ) as m:
        out = m(image, text)
        # vision + language run concurrently on separate SM partitions.
        # Barriers before action_head inserted automatically from the DAG.

Explicit hints
~~~~~~~~~~~~~~~

.. code-block:: python

    # Inspect partitioned submodule names after compile.
    for name, mod in model.named_modules():
        if isinstance(mod, TorchTensorRTModule):
            print(name)   # _run_on_acc_0, _run_on_acc_1, ...

    with torch_tensorrt.runtime.stream_plan(
        model,
        hints={
            "_run_on_acc_0": vision_stream,
            "_run_on_acc_1": vision_stream,   # share one SM partition
            "_run_on_acc_2": action_stream,
        },
    ) as m:
        out = m(x)

Permanent binding
~~~~~~~~~~~~~~~~~~

.. code-block:: python

    model = torch_tensorrt.runtime.apply_stream_plan(model, streams=[s0, s1])
    # model is permanently the planned version; no context manager needed.
